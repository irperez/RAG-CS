# RAG Server Development - Project Scope Summary

The RAG-CS repository implements a specialized Retrieval-Augmented Generation (RAG) server using a 100% local stack and modern C# patterns. Agent should be aware of the following architectural constraints and workflows.

## 📂 Core Architecture
*   **Primary Function:** The system aims to answer questions based on proprietary local documents (Obsidian Vault).
*   **Key Services:** The application structure heavily relies on the following components/services:
    *   `DocumentMonitorService`: The active background worker responsible for monitoring the network share for file changes and triggering re-ingestion.
    *   `RagIngestionEngine`: The orchestration layer that controls the entire chunking, embedding, and upserting process.
    *   `RAGQueryService`: The primary user-facing service for handling search queries (Query $\rightarrow$ Embed $\rightarrow$ Search $\rightarrow$ Generate).
*   **Dependencies:** `Program.cs` wires all services via Dependency Injection (DI) and must host the `DocumentMonitorService` as a background task.

## 🔄 Critical Workflows

### 1. Document Ingestion Flow (Background/Triggered)
1.  **Source Reading:** Use `ObsidianFileReader` to read content and calculate content hashes for change detection.
2.  **Processing:** `RagIngestionEngine` $\rightarrow$ `TextChunker` splits content into overlapping chunks.
3.  **Embedding:** Chunks are sent to `OllamaEmbeddingGenerator` for vectorization (must use `HttpClient` abstraction for connection).
4.  **Storage:** Vectors and metadata (source file, citation links) are batch-upserted into the `QdrantVectorStore`.

### 2. Query Flow (Runtime)
1.  **Vectorization:** The user query must first be embedded using `IEmbeddingGenerator` to generate a query vector.
2.  **Retrieval:** `IVectorStore` queries Qdrant with the vector to retrieve top-K relevant context chunks.
3.  **Generation:** `RAGQueryService` must assemble a prompt (Context + Query) and call the local LLM endpoint (`gemma:instruct`) to generate the final answer and citations.

## ⚠️ Operational Gotchas & Constraints
*   **Local Stack:** The entire stack (Embedding model, LLM) is confined to the local machine using Ollama. Treat the service endpoints as critical dependencies.
*   **State Management (CRITICAL):** The `DocumentMonitorService` currently uses an *in-memory* state (`Dictionary) for `lastKnownMetadata`. For production readiness, this state *must* be persisted to a reliable data store (e.g., Redis) after every ingestion run.
*   **Citation Parsing:** The `ObsidianFileReader` needs a dedicated structural parser to fully extract and persist citation link metadata (e.g., `[[Wikilinks]]`) alongside raw text.
